# E2E Test Stack

Local, resettable end-to-end regression suite for the PWA. Drives the real app in a
browser (Playwright) against a real local backend (`wrangler dev`) and a real local
database (Supabase CLI). No cloud, no production data, no CI secrets.

Covers the two primary journeys: **add-expense** and **view-summary**, plus an
environment smoke check.

## Prerequisites (one-time)

1. **Docker** running, with WSL2 integration enabled. Verify: `docker info`.
2. **Supabase CLI** — installed as a backend dev-dependency (invoke via `pnpm exec supabase`).
   If missing: `pnpm -C backend add -D supabase`.
3. **Playwright Chromium**: `pnpm -C e2e install && pnpm -C e2e exec playwright install chromium`.

## One-time local setup

```bash
cd backend
pnpm exec supabase start                  # boots Postgres + PostgREST (Docker)
pnpm exec supabase db reset               # applies migrations + seed.sql (137-row catalog) + grants
bash scripts/gen-e2e-dev-vars.sh          # writes .dev.vars.e2e (service-role key pulled from supabase status)
```

`supabase start` is a kept-running prerequisite — leave it up between runs. Re-run
`supabase db reset` only after a schema or category-snapshot change (the suite resets
*transactional* data itself, before every test).

## Run

```bash
cd e2e
pnpm test                 # Playwright auto-starts wrangler (:8787) + Vite (:5300), runs all specs
pnpm run test:headed      # watch the browser
pnpm run test:ui          # Playwright UI mode
pnpm run report           # open the last HTML report
pnpm run typecheck        # tsc --noEmit
```

Expected: **7 passing** in a few seconds.

## How it works

- **`playwright.config.ts`** — two `webServer` entries auto-start the backend
  (`wrangler dev --env e2e`) and the PWA (Vite on **5300**; 5173 is inside the WSL2-blocked
  5144–5243 range). Tests run serially (`workers: 1`) because they share one database.
- **`fixtures/test.ts`** — the shared `test`: a `beforeEach` that calls `resetDb()` and a
  `page` that seeds the auth key before app code runs.
- **`fixtures/reset-db.ts`** — truncates the transactional tables and re-inserts the
  baseline (preserves `categories`), so every test starts from the identical state.
- **`fixtures/baseline.ts`** — the deterministic baseline transactions and their known
  aggregates (grand 410; 食 350, 行 60).
- **`fixtures/auth.ts`** — `TEST_API_KEY = 'e2e-test-key'` (matches backend `ANDROID_API_KEY`),
  seeded into `localStorage['expense_api_key']`.

Backend env for `--env e2e` lives in the gitignored `backend/.dev.vars.e2e` (generated by
`scripts/gen-e2e-dev-vars.sh`; template: `.dev.vars.e2e.example`). No key is ever committed.

## Verifying the net catches regressions (SC-004)

The suite should go red when a covered flow breaks:

1. Temporarily break a flow — e.g., in `pwa/src/screens/EntryScreen.tsx` change the
   success toast text `記錄成功！`, or disable the submit guard.
2. `cd e2e && pnpm test` → the matching spec **fails**.
3. Revert → green again.

## Troubleshooting

- **`ECONNREFUSED` :54321** — `supabase start` isn't running.
- **401s** — `backend/.dev.vars.e2e` missing/stale; re-run `bash scripts/gen-e2e-dev-vars.sh`.
- **`permission denied for table …`** — `seed-grants.sql` didn't load; re-run `supabase db reset`.
- **Empty category picker** — `seed.sql` not generated; `pnpm -C backend exec tsx supabase/seed/build-seed.ts` then `supabase db reset`.
- **Port 5300 in use / page won't load** — keep Vite on 5300 (avoid 5144–5243).
