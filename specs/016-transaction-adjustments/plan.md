**Branch**: `016-transaction-adjustments` | **Date**: 2026-05-24 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/016-transaction-adjustments/spec.md`

---

## Summary

Add a `transaction_adjustments` table (kind: fee | refund | discount) to record order-level monetary modifiers, and a `transaction_items.effective_amount` column that distributes the paid total proportionally across items. Rewrite the summary RPC to aggregate on `effective_amount` so category drill-in reflects actually-paid amounts rather than MSRP. Run a one-time migration script to clean up all anomalies surfaced by the 015 audit: 15,157 category tags on the wrong table, 6 fee/refund transaction rows to convert, 6 transactions missing items, 24 orphaned item tags, and 2 item-sum mismatches. Update the PWA entry/edit flow to accept adjustments. Patch the audit script with the new data model.

### Phase B — TypeScript types + DB helpers

**B1**: Add `TransactionAdjustment` interface to `backend/src/types.ts`

**B2**: Add `effective_amount: number | null` to `TransactionItemRow` in `backend/src/types.ts`

**B3**: Add to `backend/src/db/queries.ts`:
- `insertAdjustments(supabase, transactionId, adjustments[])` — bulk insert into `transaction_adjustments`
- `computeAndWriteEffectiveAmounts(supabase, transactionId, paidTotal)` — reads items, computes floor+remainder distribution, bulk-updates `effective_amount`
- `getAdjustmentsForTransaction(supabase, transactionId)` — fetch adjustments (needed for edit flow)

**Distribution algorithm** (in `computeAndWriteEffectiveAmounts`):
```typescript
const eligible = items.filter(i => i.amount != null);
const itemsTotal = eligible.reduce((s, i) => s + i.amount!, 0);
if (itemsTotal === 0) return; // edge case: all null
const shares = eligible.map(i => ({ id: i.id, ea: Math.floor(i.amount! * paidTotal / itemsTotal) }));
const remainder = paidTotal - shares.reduce((s, x) => s + x.ea, 0);
// Add remainder to largest item (ties: last by sort_order)
const largest = eligible.reduce((a, b) => b.amount! >= a.amount! ? b : a);
const largestShare = shares.find(s => s.id === largest.id)!;
largestShare.ea += remainder;
// Bulk update
await supabase.from('transaction_items').upsert(shares.map(s => ({ id: s.id, effective_amount: s.ea })));
```

### Phase C — Data cleanup migration script

Write `backend/scripts/migrate-016.ts`. Supports `--dry-run` flag. Executes these steps in order:

**C1 — Backfill transactions_without_items (6 rows)**
For each transaction with no items, insert one default item: `{ name: transaction.note ?? '（未分類）', amount: transaction.amount, tags: transaction.tags.filter(t => t.includes(':')), sort_order: 0 }`. Then set `effective_amount = amount`.

**C2 — Migrate category_tag_on_transaction (15,157 rows)**
For each transaction where `tags` contains a `major:sub` tag:
1. Find its item(s) in `transaction_items`.
2. If exactly one item: add the `major:sub` tag to that item's `tags` array.
3. If multiple items: add the tag to the first item (sort_order ascending).
4. Remove the `major:sub` tag from `transactions.tags`.

**C3 — Convert fee/refund transactions → adjustments (6 rows)**
For each transaction with `parent_transaction_id IS NOT NULL` and `transaction_type IN ('refund', 'fee')`:
1. Insert a row into `transaction_adjustments` with:
   - `transaction_id = parent_transaction_id`
   - `kind = transaction_type` (fee → 'fee', refund → 'refund')
   - `amount = ABS(transaction.amount)`
   - `transaction_at = transaction.transaction_at`
   - `source = 'legacy_migration'`
   - `note = transaction.note`
2. Delete the original fee/refund transaction row.
3. Recompute `effective_amount` on the parent transaction.

**C4 — Print orphan fee/refund rows (6 rows, no-op)**
For transactions with `transaction_type IN ('refund', 'fee')` and `parent_transaction_id IS NULL`, print IDs + amounts to stdout. Leave untouched.

**C5 — Fix orphan_category_tag_on_item (24 rows)**
Apply the hard-coded `OLD_TAG → NEW_TAG` mapping table (authored by implementer after inspecting rows). For each item with a tag in the mapping, replace the old tag with the new one.

**C6 — Fix items_sum_mismatch (2 rows)**
For each transaction where `SUM(items.amount) ≠ transaction.amount`, inspect and correct the discrepant item amounts. (These are case-by-case; the implementer must inspect the 2 rows and determine the correct fix.) After fixing, recompute `effective_amount`.

**C7 — Final recompute pass**
For all transactions that have at least one `transaction_adjustments` row (after C3), re-run `computeAndWriteEffectiveAmounts` to ensure the `effective_amount` distribution accounts for the adjustments.

### Phase F — Audit script patches

**F1**: Rewrite the `items_sum_mismatch` check in `backend/scripts/audit-legacy.ts`
Old: `SUM(items.amount) ≠ transaction.amount`
New: `SUM(items.effective_amount) ≠ transaction.amount` (for transactions with effective_amount populated). For transactions with no items having effective_amount, fall back to the old check.

**F2**: Add new invariant check:
`SUM(items.effective_amount) + SUM(adj.amount WHERE kind = 'fee') - SUM(adj.amount WHERE kind IN ('refund', 'discount')) ≠ transaction.amount`

**F3**: Add heuristic pattern check for pre-016 fake-refund-as-discount rows:
`transaction_type = 'refund' AND parent_transaction_id IS NOT NULL AND (amount % parent.amount IN (5, 10, 15, 20 pct) OR amount % 100 = 0) AND ABS(transaction_at - parent.transaction_at) < 5 minutes`

## Key constraints (surgical change reminders)

- Do NOT modify `transactions.amount` on any existing row — it is always authoritative.
- Do NOT add triggers or generated columns.
- The migration script must be idempotent (re-runnable without duplicating rows). Use `ON CONFLICT DO NOTHING` or existence checks.
- All `effective_amount` computation happens via `computeAndWriteEffectiveAmounts`; never compute inline.
- SC-009: migration must filter to `source = 'legacy_migration'` for the category-tag migration to avoid touching non-legacy rows.
