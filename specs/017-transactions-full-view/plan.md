## Summary

Add a `v_transactions_full` read-only SQL VIEW to Supabase that joins `transactions`, `transaction_items`, and `transaction_adjustments` into one row per transaction, with `items` and `adjustments` as embedded JSON arrays. Single migration file, zero app code changes.

**Target Platform**: Supabase PostgreSQL

**Constraints**: Read-only; no app code changes; correlated subqueries to avoid cross-product fan-out
**Scale/Scope**: ~15k transactions, single user

## Phase A — Write Migration

File: `backend/supabase/migrations/017_transactions_full_view.sql`

Use correlated subqueries (not LEFT JOIN + GROUP BY) to avoid cross-product fan-out when a transaction has both items and adjustments.

```sql
CREATE OR REPLACE VIEW v_transactions_full AS
SELECT
  t.id,
  t.amount,
  t.transaction_type,
  t.payment_method,
  t.tags,
  t.note,
  t.transaction_at,
  t.created_at,
  t.updated_at,
  t.parent_transaction_id,
  t.source,
  COALESCE(
    (SELECT json_agg(
       jsonb_build_object(
         'id',               ti.id,
         'name',             ti.name,
         'amount',           ti.amount,
         'effective_amount', ti.effective_amount,
         'tags',             ti.tags,
         'sort_order',       ti.sort_order
       ) ORDER BY ti.sort_order
     )
     FROM transaction_items ti
     WHERE ti.transaction_id = t.id
    ), '[]'::json
  ) AS items,
  COALESCE(
    (SELECT json_agg(
       jsonb_build_object(
         'id',          ta.id,
         'kind',        ta.kind,
         'amount',      ta.amount,
         'note',        ta.note,
         'basis',       ta.basis,
         'basis_value', ta.basis_value,
         'source',      ta.source
       ) ORDER BY ta.created_at
     )
     FROM transaction_adjustments ta
     WHERE ta.transaction_id = t.id
    ), '[]'::json
  ) AS adjustments
FROM transactions t;
```
