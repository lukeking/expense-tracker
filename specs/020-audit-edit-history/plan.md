# Implementation Plan: Audit Edit History

**Branch**: `020-audit-edit-history` | **Date**: 2026-05-30 | **Spec**: [spec.md](spec.md)

## Summary

Every successful expense edit is recorded in a new `transaction_edit_history` table as a JSONB diff (before/after values). The diff is computed app-side in the PUT handler before the update executes. History is returned in the existing GET endpoint and rendered as a collapsible section at the bottom of `EditExpenseSheet`.

## Technical Context

**Language/Version**: TypeScript (strict), Node-compatible CF Workers runtime  
**Primary Dependencies**: Hono (routing), Supabase JS client, React 18, TanStack Query v5  
**Storage**: Supabase PostgreSQL — one new table `transaction_edit_history`  
**Testing**: Manual smoke test (quickstart.md); no automated test suite for handlers  
**Target Platform**: Cloudflare Workers (backend) + Vite PWA (frontend)  
**Performance Goals**: History insert adds one extra SELECT + one INSERT to the PUT path — well within CF Workers CPU budget  
**Constraints**: CF Workers 128 MB memory / 10 ms CPU limit; no breaking changes to GET response shape (additive only)  
**Scale/Scope**: Single-user personal tool; tens of edits per month max

## Constitution Check

- [x] **I. Simplicity-First** — One new table, one helper function, one new React component. No new abstractions or projects.
- [x] **II. Offline-First on Android** — N/A; this feature is PWA-only.
- [x] **III. Serverless Boundary Compliance** — History write is a single INSERT after the existing multi-step PUT. No long-running operations.

- [x] **V. Security at System Boundaries** — No new secrets. History is accessed only through the CF Worker (same service-role key pattern as all other Supabase access).

## Implementation Phases

### Phase A — Migration

**A1** Write `backend/supabase/migrations/019_transaction_edit_history.sql` using `contracts/schema-ddl.sql`.

**A2** Apply migration to dev Supabase instance (`npx supabase migration up` or Studio SQL editor).

**A3** Verify table exists with correct columns and index via `\d transaction_edit_history`.

---

### Phase B — Backend: diff helper + PUT handler

**B1** Add `computeEditDiff` helper function in `pwa.ts` (or a small dedicated file):
- Accepts `before` snapshot and `after` payload, both typed.
- Returns the diff object or `null` if no changes.
- Implements normalisation rules from `data-model.md` (note null/empty equivalence, tag sort, item/adjustment deep equality).

**B2** Extend the PUT handler (`pwa.ts`) — before the existing `supabase.from('transactions').update(...)` call:
1. Read the current state: re-use the existing `SELECT` (already fetches `id, transaction_type, transaction_at`) — extend it to also fetch `amount, payment_method, tags, note`.
2. Read current items: `SELECT id, name, amount, tags, note FROM transaction_items WHERE transaction_id = txId ORDER BY sort_order`.
3. Read current adjustments: call existing `getAdjustmentsForTransaction(supabase, txId)`.
4. Call `computeEditDiff(before, after)`.
5. If diff is non-null, insert a row into `transaction_edit_history`.

> History insert happens AFTER the update succeeds to avoid writing history for a failed save. Order: validate → update header → replace items → replace adjustments → recompute effective amounts → insert history (only if all prior steps succeeded).

**B3** Extend `GET /pwa/transactions/:id` to also query `transaction_edit_history`:
```sql
SELECT id, edited_at, diff
FROM transaction_edit_history
WHERE transaction_id = $txId
ORDER BY edited_at ASC
```
Append result as `history: []` in the JSON response.

---

### Phase C — Frontend

**C1** Update `TxDetail` type in `EditExpenseSheet.tsx` to add:
```ts
history: { id: string; edited_at: string; diff: EditDiff }[];
```
Define `EditDiff` type locally (mirrors the JSONB schema from `data-model.md`).

**C2** Create `pwa/src/components/EditHistorySection.tsx`:
- Props: `history: TxDetail['history']`
- Renders nothing when `history.length === 0`.
- Collapsed by default; header shows "編輯紀錄 (N)" with expand toggle.
- Each entry: `edited_at` formatted as `YYYY-MM-DD HH:mm` (local time), followed by a diff summary:
  - Header fields: one line per changed key — label (Chinese) + "before → after".
  - Items: if present, "items: M → N 項" (count change).
  - Adjustments: if present, "折抵: M → N 筆".

**C3** Render `<EditHistorySection history={tx.history} />` at the bottom of `EditExpenseFormInner`, below the Note field and above the Save button.

---

### Phase D — Smoke test

Follow `quickstart.md` Steps 1–6. Verify:
- History absent for unedited transaction.
- History entry created after save with correct diff.
- No-op save does not create entry.
- Multiple edits accumulate in order.
