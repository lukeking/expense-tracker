# Implementation Plan: Auto-Create Item from Subcategory

**Branch**: `007-fix-item-note-clarity` | **Date**: 2026-05-10 | **Spec**: [spec.md](./spec.md)  
**Input**: Feature specification from `specs/007-item-from-subcategory/spec.md`

## Summary

When a Discord `/expense` command description contains a `#category:subcategory` tag but no explicit item tokens, the expense parser auto-creates a single item `{ name: subcategoryName, amount: totalExpenseAmount }`. This eliminates the redundant typing required today (e.g., `#food:餐飲, 餐飲 300`). The change is entirely within `parseDescription()` in `expense-parser.ts` — no schema changes, no new dependencies, no new files.

## Phase 0: Research

No `NEEDS CLARIFICATION` items exist. One edge-case decision resolved here:

### Decision: Empty subcategory string (`#food:`)

- **Chosen**: Do NOT auto-create when subcategory is an empty string. Guard: `subcategoryName.trim().length > 0`.
- **Rationale**: An item with an empty name is useless as a record and would be confusing in the confirmation message. The user almost certainly made a typo.
- **Alternative**: Auto-create with name="" — rejected; bad data.

### Decision: Subcategory with spaces (`#food:牛肉 麵`)

- **Chosen**: Use the raw subcategory string including spaces. Tokens are comma-separated, so `#food:牛肉 麵` is a valid single token and the subcategory is `牛肉 麵`.
- **Rationale**: Consistent with how the rest of the parser handles token content; no special stripping needed.

## Phase 1: Design

### data-model.md

The auto-created item follows the identical `InvoiceItem`-like shape already used throughout:

```
Item { name: string, amount: number }
```

**Auto-creation rule** (codified):

```
IF parsed.items.length === 0
AND parsed.categoryTag is non-null
AND subcategoryName (part after ':') has length > 0
THEN items = [{ name: subcategoryName, amount: totalExpenseAmount }]
```

Where `subcategoryName = categoryTag.split(':')[1]` (second segment, trimmed).

**Trigger**: Discord `/expense` command only, inside `parseDescription()` in `expense-parser.ts`. The function already receives `totalAmount` as its second argument.

**No DB migration**: The `items` JSONB column in `transactions` already stores `{ name, amount }` arrays. Auto-created items are structurally identical to manual items.
