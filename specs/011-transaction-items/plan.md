# Implementation Plan: Transaction Items Table

**Branch**: `011-transaction-items` | **Date**: 2026-05-16 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/011-transaction-items/spec.md`

## Summary

Normalise transaction line items from a JSONB column on `transactions` into a dedicated `transaction_items` table. Each item gets its own optional amount and category tags, enabling per-item category breakdown in `/summary`. The `transactions` record remains the payment envelope (total amount, payment method, store-level tags). Existing test data is dropped; no backward migration required.

## Implementation Notes

### Key invariant: transaction.amount is always the ground truth

`SUM(transaction_items.amount)` is informational, not authoritative. The transaction total is the paid amount. Item amounts are the user's breakdown of how that total was spent. Partial splits (amounts sum to less than total) are valid — the gap falls to `其他`.

### Summary aggregation algorithm (replaces current `aggregateByCategory`)

```
Input: transactions[] each with nested transaction_items[]

For each transaction t:
  categorisedSum = 0
  for each item i in t.transaction_items:
    if i.amount is not null and i.tags contains a 'X:Y' tag:
      add i.amount to category i.tags[0].split(':')[0]
      categorisedSum += i.amount
  remainder = t.amount - categorisedSum
  if remainder > 0: add remainder to '其他'

Aggregate across all transactions → sorted category totals
```

### /expense parser extension (expense-parser.ts)

Extended token grammar — a comma-token that begins with `#category:subcategory` applies that tag to the item in the same token:

```
"#食:早餐 便當 60"  →  item { name:"便當", amount:60, tags:["食:早餐"] }
"#食:午餐"          →  implicit item from subcategory name (existing behaviour)
"#全家"             →  plain transaction tag (no colon — unchanged)
```

### Write sequence: transaction then items

1. `insertTransaction(supabase, txData)` → returns transaction with id
2. `insertTransactionItems(supabase, transactionId, items)` → bulk insert into `transaction_items`

Partial failure (step 1 succeeds, step 2 fails): transaction recorded at correct total, items missing → full amount falls to `其他`. Acceptable for a personal tool. User can see missing items and re-enter.

### /amend cascade rules

Read `transaction_items` for the target transaction before updating:
- Count items, check their amounts
- Single item matching old amount → cascade both
- Multiple items or mismatch → warn, update transaction only
- No item amounts → update transaction only, no warning

### Invoice import item write (existing reconcile/import flow)

After matching invoice → transaction:
- Fetch `transaction_items` for the matched transaction
- If any items have `amount IS NULL` → populate from `invoice.items` in order
- If item count differs → replace all transaction items with invoice items
- If all items already have amounts → skip (do not overwrite)
